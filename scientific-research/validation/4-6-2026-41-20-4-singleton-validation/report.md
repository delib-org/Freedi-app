## 1. Objective

Test that a **single statement that belongs to neither cluster** stays
**unclustered** (archetype C "lone singleton" / F "noise"): the pipeline should
recover the real synths and **not force the lone statement into one of them**.

## 5. The bug this run found and fixed

The **first** run of this corpus FAILED here (14/18): the off-topic statement was
**absorbed into the `daily-cardio` synth** (making it 5 members, 80% pure).
Root-causing it produced a real production fix.

- **Why it was absorbed.** DBSCAN runs in the UMAP-reduced space, where the
  off-topic point sat within ε of the `daily-cardio` core, so it was clustered
  directly (never flagged as noise). Then `twoTierJudge`'s keep tier kept it: the
  original code, when a cluster reached ≥`keepThreshold` (80%) agreement, set the
  synth's `memberIds` to the **entire** original cluster — *including* the
  cosine-auto-rejected / LLM-dissenting members. The off-topic statement (cosine
  to the medoid below the 0.60 reject band → dissent) was retained because its 4
  `daily-cardio` neighbours agreed.
- **The fix** (`functions/src/synthesis/twoTierJudge.ts`): unify the keep/split
  branches so a kept synth always contains **only the agreed subset** (medoid +
  auto-accepted + LLM-agreed); auto-rejected / dissenting members are never kept,
  regardless of how coherent the rest of the cluster is. This matched the
  already-correct behaviour of the 50–80% branch. Coherent clusters (every member
  auto-accepted, e.g. the tight corpus) are unchanged. Regression test added in
  `twoTierJudge.test.ts`.
- **Result after the fix** (this run): the cosine step still groups the off-topic
  with `daily-cardio` (cluster of 5), but the judge excludes it → `daily-cardio`
  synth = 4 pure members, off-topic left unclustered. 14/18 → **18/18**.

This is a precision improvement to the production judge (both the live and bulk
paths inherit it via `VerifiedCluster.memberIds`): off-topic / outlier members in
an otherwise-coherent cluster are no longer absorbed.
